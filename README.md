## Project overview

We evaluate both **classical** and **deep learning** recommender models across two datasets with different feedback types:

| Dataset | Feedback type | Users | Items | Interactions | Sparsity |
|---|---|---|---|---|---|
| MovieLens 1M | Explicit (ratings 1–5) | 6,040 | 3,416 | 999,611 | 95.2% |
| Last.fm | Implicit (play counts) | 1,090 | 3,646 | 52,551 | 98.7% |

Key research questions we address:

1. Do deep learning models outperform well-tuned matrix factorisation baselines?
2. Does modelling temporal sequence order (SASRec) improve over non-sequential models on Last.fm?
3. How do all models handle cold-start users (fewer than 5 training interactions)?
4. How sensitive are results to negative sampling ratio and strategy (implicit feedback)?

## Datasets

Both datasets are preprocessed by Role A using a consistent pipeline:

- **Deduplication** — keep the most recent interaction per user-item pair
- **5-core filtering** — retain only users and items with ≥ 5 interactions
- **ID remapping** — contiguous integer IDs `[0, N)` required for embedding layers
- **Temporal train/val/test split** — 80 / 10 / 10 per user's chronological sequence (no random split, no data leakage)
- **Binarisation** — MovieLens: `rating ≥ 4` → relevant; Last.fm: all interactions → relevant

The processed CSV files are already in `data/splits/`. You do **not** need to download or re-process the raw data to run the experiments.

## Models

### Role B — classical baselines

| Model | Dataset | Notes |
|---|---|---|
| Popularity | Both | Non-personalised; sanity check baseline |
| SVD (Matrix Factorisation) | MovieLens | Via `Surprise` library, explicit feedback |
| ALS (Matrix Factorisation) | Last.fm | Via `implicit` library, implicit feedback |

### Role C — deep learning models

| Model | Dataset | Description |
|---|---|---|
| NeuMF | Both | GMF branch + MLP branch fused at output (He et al., 2017) |
| Two-tower | Both | Separate user/item encoders, dot-product similarity; standard candidate-generation architecture |
| SASRec | Last.fm only | Self-attentive Transformer over user interaction history (Kang & McAuley, 2018) |

All deep learning models are trained with:
- Hyperparameters selected via **Optuna** (TPE sampler, 25–30 trials) on the validation set
- Results reported as **mean ± std across 3 random seeds** (42, 123, 2024)
- Early stopping with patience = 5 on validation loss

## Quickstart

### 1. Clone the repository

```bash
git clone https://github.com/<your-org>/recsys-project.git
cd recsys-project
```

### 2. Install dependencies

```bash
# Role C dependencies
pip install -r role_C/requirements.txt

# Role B dependencies (if applicable)
# pip install -r role_B/requirements.txt
```

> Python 3.9+ recommended. GPU is optional but significantly speeds up SASRec training.

### 3. Verify data is in place

```bash
ls data/splits/movielens-1m/   # train.csv  val.csv  test.csv  metadata.json
ls data/splits/lastfm/         # train.csv  val.csv  test.csv  metadata.json
```

### 4. Run deep learning experiments

```bash
cd role_C/

# MovieLens 1M — NeuMF + Two-tower
python run_movielens.py

# Last.fm — NeuMF + Two-tower + SASRec
python run_lastfm.py

# Negative sampling sensitivity analysis (Last.fm)
python neg_sampling_analysis.py
```

Prediction files are written to `predictions/` in the repo root. Role D's evaluation scripts read from there.

## Reproducing results

All experiments are deterministic given the fixed seeds. To reproduce from scratch:

```bash
# Step 1 — run classical baselines (Role B)
cd role_B && python run_baselines.py

# Step 2 — run deep learning models (Role C)
cd role_C && python run_movielens.py && python run_lastfm.py

# Step 3 — evaluate all models (Role D)
cd role_D && python evaluate.py
```

Intermediate files:

| File | Description |
|---|---|
| `predictions/<dataset>/<model>.pkl` | `{user_id: [item_id_1, …, item_id_10]}` |
| `predictions/<dataset>/<model>_all_seeds.pkl` | List of 3 per-seed prediction dicts |
| `role_C/results/*_best_params.json` | Optuna-selected hyperparameters |
| `role_C/results/*_history_seed*.json` | Training/validation loss curves |
| `role_C/results/neg_sampling_analysis.json` | Negative sampling experiment summary |

## Evaluation protocol

All models are evaluated on the **held-out test set only**, using the temporal split described above. Hyperparameter selection uses the validation set exclusively.

**Metrics** (K = 10 as primary cutoff):

| Metric | Description |
|---|---|
| HR@10 | Hit Rate — fraction of users with a relevant item in top 10 |
| NDCG@10 | Normalised Discounted Cumulative Gain — rewards higher-ranked relevant items |
| MAP | Mean Average Precision across users |

For MovieLens (explicit feedback), relevance is defined as `rating ≥ 4`.  
For Last.fm (implicit feedback), all test interactions are treated as relevant.

**Cold-start evaluation** — users with fewer than 5 training interactions are evaluated separately to assess how each model handles data-sparse users.

**Negative sampling** (implicit feedback only) — training uses uniform random negative sampling at a 1:4 positive-to-negative ratio by default. A sensitivity analysis over ratios `{1, 4, 10}` and strategies `{uniform, popularity}` is included in `role_C/neg_sampling_analysis.py`.
